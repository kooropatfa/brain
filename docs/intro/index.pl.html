Problem

Firma ciągle zapomina — i nasz AI też

Najmądrzejsze rzeczy, które wiemy, siedzą w głowach, martwych wątkach na Slacku i spotkaniach, których nikt nie spisał. Gdy ktoś zapomni — albo odejdzie — to przepada. Każda sesja Claude'a też startuje od zera.

Szczegóły

Odpowiadamy na te same pytania, odbudowujemy ten sam kontekst i od nowa kłócimy się o ustalone decyzje. Koszt to nie jeden zgubiony dokument — to narastający podatek firmy, która nie pamięta samej siebie, płacony co tydzień przez wszystkich.

Technicznie

LLM jest bezstanowy między sesjami; okna kontekstu nie trwają. „Pamięć” dostawcy jest per-user i serwerowa. Brakuje wspólnego, trwałego, audytowalnego magazynu, który cała firma i jej agenci czytają i zapisują. Robimy z niego repo Git z Markdownem — diff, review, historia, własność, zero lock-inu.

Model myślowy

Dwie połowy: wiedza i silnik

Jedyna rzecz do zapamiętania. Wiedza, którą posiadasz żyje w Twoim repo. Silnik, który nią porusza to jeden zainstalowany plugin, który nie trzyma żadnej wiedzy.

📚 Wiedza — Twoje repo
co wiesz · jedno na zespół/projekt
  • czyste notatki .md, wg wymiaru
  • decyzje (ADR-y) + słownik
  • prywatne, czyste dane — bez silnika w środku
  • czytelne na zawsze, przez wszystko
⇄
⚙️ Silnik — plugin
jak się porusza · instalowany raz
  • kooropatfa/brain — plugin Claude Code

  • auto-update; zero wiedzy w środku

Szczegóły

Po co je dzielić? Wiedza musi przeżyć każde narzędzie, które jej dotyka. Skasuj plugin jutro, a wiedza dalej tam jest — czysty Markdown w Twoim Gicie, czytelny na GitHubie czy w Obsidianie.

Silnik to wymienialne rury, których nigdy nie edytujesz. Wiedza to aktywo. Rozdzielenie sprawia, że Brain jest Twój, a nie funkcją czyjegoś produktu.

Technicznie

  • Repo silnika: kooropatfa/brain — manifest pluginu, skills/brain/, tools/ (sync, init, klasyfikator, reviewer), reużywalne workflowy CI. Bez wiedzy .md.
  • Repo wiedzy: generowane przez brain-init — foldery + MOC + config, nic więcej.
  • Klon: każdy Brain żyje w ~/.brain/<name>; nazwa katalogu jest nazwą Brainu.
Jeden plugin, wiele mózgów

Multi-brain, multi-projekt — każdy projekt wybiera swój Brain

Zainstaluj plugin raz. Prowadź dowolnie wiele Brainów na jednej maszynie — praca, prywatne, per projekt. Każdy projekt deklaruje, których Brainów używa, w pliku .brains.yml; wszystkie pozostałe milczą.

Szczegóły

🏢 Projekt firmowy → Hooper Brain
Twoje firmowe repo Hooper wiąże zespołowy Brain (tonsser/hooper-brain). „wrzuć to do braina” trafia tam.
🏠 Projekty prywatne → Twój Brain
Projekt poboczny wiąże Twój prywatny Brain (np. kooropatfa/muzg-knowledge). Ten sam gest, inne repo — wiedza firmowa nigdy się nie miesza.
🔇 Ścisła izolacja
Związany projekt dotyka tylko swoich Brainów. Contribute odmawia pushu gdzie indziej; reszta milczy.
🪄 Pytanie raz
Niezwiązany projekt? Agent wypisuje Twoje Brainy, pyta raz i zapisuje odpowiedź do .brains.yml.

Technicznie

  • Zwiąż projekt: echo 'use: [hooper-brain]' > .brains.yml (nazwy = katalogi klonów w ~/.brain/; [] = brak Brainu tutaj).
  • Podłącz maszynę do Brainu: brain-sync connect --repo tonsser/hooper-brain → klonuje do ~/.brain/hooper-brain.
  • Zachowanie per-projekt przez <name>.yml + osobisty ~/.config/<name>.yml; fallback maszynowy ~/.config/brains.yml.
  • Auto-capture na koniec sesji jest bramkowany tym samym wiązaniem — nigdy nie pisze do Brainu, którego projekt nie używa.
Przepływ na jednym obrazku

Od „wrzuć to” do „agent każdego to wie”

Capture wpływa do Twojego repo wiedzy, jest klasyfikowany i składany, a następnie — domyślnie — auto-merguje się. Potem agent każdego kolegi zaciąga go przy następnej sesji.

Szczegóły

  • Wejście: „wrzuć to do braina” (albo skrót Slacka) pisze surowy capture do _inbox/.
  • Sortowanie: pipeline klasyfikuje, destyluje czystą notatkę, linkuje ją i otwiera PR Ingest:.
  • Merge (domyślnie automatyczny): reviewer strukturalny sprawdza PR i squash-merguje te czyste. Merge przez człowieka jest opcjonalny — Brain może go wymagać.
  • Wyjście: każdy agent zaciąga najnowsze na starcie sesji — i już to wie.

Technicznie

  • Capture: _inbox/<source>-<YYYY-MM-DD-HHMM>.md (schema v1).
  • Dwa tryby ingestii: local (domyślny — agent sesji klasyfikuje, bez API/CI) albo action (pipeline GitHub Actions w repo wiedzy).
  • Reviewer auto-merguje PR-y o znanym bezpiecznym kształcie; flaga niepewności lub nieudany check eskaluje do człowieka, nigdy nie merguje w ciemno.
  • Wyłącz auto-merge dla Brainu przez auto_merge: false w jego shimie review-ingest.
Karmienie

„wrzuć to do braina”

Powiedz to po ludzku i wskaż cokolwiek — PDF, maila, wątek na Slacku, screena, albo po prostu „to, co przed chwilą rozkminiliśmy”. Nie uczysz się formatu i nie wypełniasz szablonu.

„wrzuć to do braina”  →  📎 chaotyczny artefakt  ⟶  🗂️ schludna, opisana, połączona notatka

Szczegóły

🗣️ Nie-techniczni
Jedno kliknięcie „Add to the Brain” na Slacku — najedź, wybierz, gotowe.
💻 W agencie
Powiedz Claude'owi „wrzuć to do braina”. Pisze capture i otwiera PR.
🤝 Proaktywnie
Gdy pojawi się realna decyzja albo insight, agent proponuje zapis — bez męczenia.
🔚 Na koniec sesji
Opcjonalny hook automatycznie destyluje trwałą wiedzę z sesji.

Technicznie

Ścieżka agenta waliduje capture, potem brain-sync contribute otwiera PR Capture: tylko przeciw związanemu Brainowi projektu. Ścieżka Slacka = skrót wiadomości + modal → relay n8n (z weryfikacją podpisu) → GitHub Contents API. Działa też obserwowany folder („jak Dropbox”) przez inbox-watch.

Korzystanie

Czytaj jak człowiek albo pozwól AI czytać za Ciebie

Wiedza to po prostu pliki. Ludzie je przeglądają; agenci zaglądają do nich automatycznie przed nietrywialną pracą — przestają zgadywać i od nowa rozstrzygać ustalone rzeczy.

Szczegóły

👤 Ludzie

  • Otwórz jako vault Obsidian — backlinki, graf, wyszukiwarka.
  • Albo po prostu czytaj na GitHubie. Bez terminala.
  • Zmiany reviewowane jak kod: diff, autorstwo, odwracalne.

🤖 Agenci

  • Sync związanego Brainu na starcie sesji: 🧠 Brain zsynchronizowany.
  • Zaglądają do właściwego wymiaru przed pracą.
  • Nigdy nie roztrząsają decyzji, która już jest w decisions/.

Technicznie

Brak wbudowanej wyszukiwarki — agent grepuje klon w ~/.brain/<name> wprost, zaczyna od index.md każdego wymiaru (Mapa Treści), respektuje łańcuchy ADR status: i ładuje do kontekstu tylko istotny wycinek.

Czemu własny

Żadna wbudowana funkcja nie daje wspólnego, posiadanego, wersjonowanego mózgu zespołu

Claude ma funkcje pamięci — ale każda jest lokalna na laptopie, per-osoba, albo serwerową czarną skrzynką. Żadna nie jest pulą, którą zespół posiada i może reviewować.

Instalacja

Jedna instalacja na Claude Code — Mac i Windows

Cienka warstwa na Claude Code. Zainstaluj raz; plugin auto-updatuje się i obsługuje każdy Brain na maszynie.

📚 Twoje Brainy — wiedza w Twoich repo Git
⚙️ plugin brain — sync · klasyfikacja · review · PR-y (jedna instalacja, N mózgów)
Claude Code — agent, którego już używasz

Szczegóły

Instalacja (raz na maszynę):

# wymagania + logowanie w przeglądarce (wybierz OS):
curl -fsSL https://raw.githubusercontent.com/kooropatfa/brain/main/install.sh | bash   # macOS/Linux
irm https://raw.githubusercontent.com/kooropatfa/brain/main/install.ps1 | iex          # Windows

# potem, w Claude Code:
/plugin marketplace add kooropatfa/brain
/plugin install brain@brain

Stwórz Brain: brain-init scaffolduje repo wiedzy. Podłącz: brain-sync connect --repo owner/name. Zwiąż: use: [name] w .brains.yml.

Technicznie

Bez kopiowania PAT — gh auth login (przeglądarka) pokrywa push + PR. Repo wiedzy nie mają silnika, więc nie ma w nich czego aktualizować; silnik = plugin, który Claude Code aktualizuje automatycznie. W trybie action shimy w repo wiedzy wołają reużywalne workflowy silnika przypięte do ruchomego tagu @v1.

Prośba

W tym tygodniu: nakarm Brain jedną rzeczą, którą wiesz tylko Ty

Brain to pusty szkielet, dopóki go nie nakarmisz. Jest tylko tak mądry, jak to, co my w nim umieścimy.

Dodaj jedną rzecz, której nikt nie zapisał — decyzję, insight o kliencie, czemu wybraliśmy X zamiast Y, zasadę designu. Jedno zdanie do agenta. Jeden PR. Auto-merguje się, jeśli jest czysty. Firma robi się odrobinę mniej zapominalska — na zawsze.

Szczegóły

Nie przemyśliwuj formatu — nie ma czego się uczyć. Jeśli warto, by kolega wiedział to za trzy miesiące, warto to zapisać. Pipeline składa; reviewer merguje; cały zespół ma dostęp.

Technicznie

domyślnie  Czyste PR-y ingestii auto-mergują się (reviewer strukturalny). opcjonalnie  Brain może wymagać człowieka przy każdym merge przez auto_merge: false. Ingest trwa chwilę — nigdy natychmiast.

Brain jest dla firmy tym, czym AGENTS.md dla repo — ale żywy, uporządkowany wg wymiarów, współdzielony przez wszystkich i ciągle karmiony. Karmisz go jednym zdaniem: „wrzuć to do braina”.
