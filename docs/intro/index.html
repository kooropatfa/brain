The problem

The company keeps forgetting — and so does our AI

The smartest things we know live in heads, dead Slack threads, and meetings nobody wrote up. When someone forgets — or leaves — it's gone. Every Claude session starts from zero, too.

Details

We re-answer the same questions, rebuild the same context, and re-argue settled decisions. The cost isn't one lost doc — it's the compounding tax of a company that can't remember itself, paid every week by everyone.

Technical

An LLM is stateless across sessions; context windows don't persist. Vendor "memory" is per-user and server-side. What's missing is a shared, durable, inspectable store the whole org and its agents read and write. We make it a Git repo of Markdown — diff, review, history, ownership, zero lock-in.

The mental model

Two halves: the knowledge and the engine

The one idea to take away. The knowledge you own lives in your repo. The engine that moves it is one installed plugin that holds no knowledge at all.

📚 Knowledge — your repo
what you know · one per team/venture
  • plain .md notes, by dimension
  • decisions (ADRs) + a glossary
  • private, pure data — no engine inside
  • readable forever, by anything
⇄
⚙️ Engine — the plugin
how it moves · installed once
  • kooropatfa/brain — the Claude Code plugin
  • brain-sync · brain-init · classifier · reviewer
  • serves any number of Brains on a machine
  • auto-updates; zero knowledge inside

Details

Why split them? The knowledge must outlive every tool that touches it. Delete the plugin tomorrow and the knowledge is still there — clean Markdown in your Git, readable on GitHub or in Obsidian.

The engine is swappable plumbing you never edit. The knowledge is the asset. Separation is what makes a Brain yours, not a feature of someone's product.

Technical

  • Engine repo: kooropatfa/brain — plugin manifest, skills/brain/, tools/ (sync, init, classifier, reviewer), reusable CI workflows. No .md knowledge.
  • Knowledge repo: generated by brain-init — folders + MOC + config, nothing else.
  • Clone: each Brain lives at ~/.brain/<name>; the basename is the Brain's name.
One plugin, many brains

Multi-brain, multi-project — each project picks its Brain

Install the plugin once. Run as many Brains as you like on one machine — work, private, per venture. Every project declares which Brain(s) it uses in a .brains.yml file; all the others stay completely silent.

Details

🏢 Work project → Hooper Brain
Your professional Hooper repo binds the team Brain (tonsser/hooper-brain). "add this to the brain" lands there.
🏠 Private projects → your Brain
A side project binds your private Brain (e.g. kooropatfa/muzg-knowledge). Same gesture, different repo — work knowledge never leaks in.
🔇 Strict isolation
A bound project only ever touches its own Brain(s). Contribute refuses to push elsewhere; the rest stay silent.
🪄 Asked once
An unbound project? The agent lists your Brains, asks once, and saves the answer to .brains.yml.

Technical

  • Bind a project: echo 'use: [hooper-brain]' > .brains.yml (names = clone basenames under ~/.brain/; [] = no Brain here).
  • Connect a machine to a Brain: brain-sync connect --repo tonsser/hooper-brain → clones to ~/.brain/hooper-brain.
  • Per-project behavior via <name>.yml + personal ~/.config/<name>.yml; machine-wide fallback ~/.config/brains.yml.
  • SessionEnd auto-capture is gated by the same binding — it never writes to a Brain a project doesn't use.
The flow, in one picture

From “add this” to “everyone's agent knows it”

A capture flows into your knowledge repo, gets classified and filed, and — by default — auto-merges. Then every teammate's agent pulls it at the next session.

Details

  • In: "add this to the brain" (or the Slack shortcut) writes a raw capture into _inbox/.
  • Sort: the pipeline classifies, distils a clean note, cross-links it, and opens an Ingest: PR.
  • Merge (auto by default): a structural reviewer checks the PR and squash-merges the clean ones. Human merge is optional — a Brain can require it.
  • Out: every agent pulls the latest at session start, and now it just knows.

Technical

  • Capture: _inbox/<source>-<YYYY-MM-DD-HHMM>.md (schema v1).
  • Two ingestion modes: local (default — the session agent classifies, no API/CI) or action (a GitHub Actions pipeline in the knowledge repo).
  • Reviewer auto-merges PRs matching the known-safe shape; a confidence flag or failed check escalates to a human, never merges blind.
  • Opt a Brain out of auto-merge with auto_merge: false in its review-ingest shim.
Feeding it

“add this to the brain”

Say it in plain words and point at anything — a PDF, an email, a Slack thread, a screenshot, or just "that thing we just figured out." You learn no format and fill no template.

“add this to the brain”  →  📎 a messy artifact  ⟶  🗂️ a tidy, filed, linked note

Details

🗣️ Non-technical
A one-click “Add to the Brain” in Slack — hover a message, pick it, done.
💻 In your agent
Tell Claude "add this to the brain." It writes the capture and opens the PR.
🤝 Proactive
When a real decision or insight surfaces, your agent offers to save it — without nagging.
🔚 On session end
Optional hook distils a session's durable knowledge automatically.

Technical

The agent path validates the capture, then brain-sync contribute opens a Capture: PR against the project's bound Brain only. The Slack path = a message shortcut + modal → an n8n relay (signature-verified) → the GitHub Contents API. A watched folder ("Dropbox feel") via inbox-watch also works.

Using it

Read it like a human, or let your AI read it for you

The knowledge is just files. People browse them; agents consult them automatically before non-trivial work — so they stop guessing and stop re-deciding settled things.

Details

👤 Humans

  • Open it as an Obsidian vault — backlinks, graph, search.
  • Or just read it on GitHub. No terminal to read.
  • Changes are reviewed like code: diffed, attributed, reversible.

🤖 Agents

  • Sync the bound Brain at session start: 🧠 Brain synced.
  • Consult the right dimension before working.
  • Never re-litigate a decision already in decisions/.

Technical

No built-in search — the agent greps the clone at ~/.brain/<name> directly, starts from each dimension's index.md (a Map-of-Content), honours ADR status: chains, and loads only the relevant slice into context.

Getting it

One install on Claude Code — Mac & Windows

A thin layer on top of Claude Code. Install once; the plugin auto-updates and serves every Brain on the machine.

📚 Your Brain(s) — knowledge in your own Git repo(s)
⚙️ brain plugin — sync · classify · review · file PRs (one install, N brains)
Claude Code — the agent you already use

Details

Install (once per machine):

# prerequisites + browser sign-in (pick your OS):
curl -fsSL https://raw.githubusercontent.com/kooropatfa/brain/main/install.sh | bash   # macOS/Linux
irm https://raw.githubusercontent.com/kooropatfa/brain/main/install.ps1 | iex          # Windows

# then, inside Claude Code:
/plugin marketplace add kooropatfa/brain
/plugin install brain@brain

Create a Brain: brain-init scaffolds a knowledge repo. Connect: brain-sync connect --repo owner/name. Bind: use: [name] in .brains.yml.

Technical

No PAT to copy — gh auth login (browser) covers push + PR. Knowledge repos contain no engine, so there's nothing in them to update; the engine = the plugin, which Claude Code updates automatically. In action mode the knowledge repo's shim workflows call the engine's reusable workflows pinned to the moving @v1 tag.

The ask

This week: feed your Brain one thing only you know

A Brain is an empty skeleton until it's fed. It's only as smart as what we put in.

Add one thing nobody else has written down — a decision, a customer insight, why we chose X over Y, a design principle. One sentence to your agent. One PR. It auto-merges if it's clean. The company gets a little less forgetful — forever.

Details

Don't overthink the format — there isn't one to learn. If it's worth a teammate knowing in three months, it's worth a capture. The pipeline files it; the reviewer merges it; the whole team gets the recall.

Technical

default  Clean ingestion PRs auto-merge (structural reviewer). optional  A Brain can require a human on every merge with auto_merge: false. Ingestion takes a moment — never instant.

A Brain is to a company what AGENTS.md is to a repo — but living, organized by dimension, shared by everyone, and continuously fed. You feed it in one sentence: “add this to the brain.”
